## 2024-05-24 - API Exception Information Exposure
**Vulnerability:** The PHP API endpoint (`api.php`) was returning database exception details directly to the client via JSON responses (e.g., `echo json_encode(['error' => 'Database connection failed: ' . $e->getMessage()]);`). This could leak database connection strings, table names, syntax errors, or server details to malicious actors.
**Learning:** Exception details must never be passed to the client. This exposes internal database architecture or other infrastructure secrets which could aid in planning targeted attacks.
**Prevention:** Catch exceptions and log the detailed message server-side using `error_log()`. Return generic, user-friendly error messages such as "Hiba történt az adatbázis csatlakozás során." to the client.
## 2024-06-19 - Session Fixation in PHP
**Vulnerability:** The application did not regenerate the session ID upon user authentication or registration.
**Learning:** Without regenerating the session ID, the application is vulnerable to Session Fixation attacks, where an attacker can set a known session ID and use it to hijack the user's session after they log in.
**Prevention:** Always call `session_regenerate_id(true);` upon privilege level changes (e.g., login, registration) to issue a new session ID and invalidate the old one.
## 2025-02-12 - [SQL Injection and XSS Vulnerabilities]
**Vulnerability:** Found `innerHTML` usage in JavaScript without sanitization, leading to DOM-based XSS, and missing input sanitization in PHP backend (though prepared statements are correctly used for database interaction, output encoding or input validation against XSS is weak, specifically with the `username` field).
**Learning:** `innerHTML` is widely used throughout the dashboard to render dynamic content, including variables that come from user inputs/database (e.g., `username`, quiz explanations, feedback). Although SQL injection is largely mitigated using PDO prepared statements (`$stmt = $pdo->prepare(...)`), there's a lack of XSS protection. The user can register an account with a payload like `<script>alert(1)</script>` for `username`, which is then rendered directly via `innerHTML`. There's also no CSRF token validation on POST endpoints in `api.php`.
**Prevention:** Implement HTML entity encoding before rendering user-supplied data in the DOM (avoiding `innerHTML` for plain text, using `textContent` instead) and sanitize inputs on the server. Implement CSRF protection.
## 2025-02-13 - Host Header Injection in Password Reset
**Vulnerability:** The `handleForgotPassword` function in `api.php` used the untrusted `$_SERVER['HTTP_HOST']` variable directly to construct password reset links. An attacker could manipulate the `Host` header to point to a malicious domain. If the victim clicks the resulting link in their email, their password reset token is sent to the attacker.
**Learning:** The `Host` header is client-controlled and cannot be trusted. Relying on it for security-sensitive operations like generating password reset links or redirects can lead to severe vulnerabilities.
**Prevention:** Validate `$_SERVER['HTTP_HOST']` against a strict allowlist of known domains, or simply use a predefined, hardcoded base URL for generating sensitive links.
